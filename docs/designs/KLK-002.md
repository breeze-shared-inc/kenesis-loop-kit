## 1. 目的

`docs/SPEC.md` §1・§4・§5 に基づき、コーポレートサイトのトップページのファーストビュー（FV）を刷新し、
「ITインフラ」「SES」の2事業を**対等の比重**（同等の面積・視覚的重み）で提示する。FV内に「お問い合わせ（サービス）」
「採用情報（SES採用）」の2CTAを並置し、訪問者・SES志望エンジニアの双方が迷わず目的の導線へ到達できる状態にする。
本チケットのスコープは **FVと2事業提示まで**（採用ページ本体・採用専用窓口・GA4本実装は別チケット）。

**前提（人間判断で2026-06-25確定。本設計はこれに従う）:**
- 実装形式は **静的HTML/CSS/JS**。SPEC §7 の「WordPress子テーマ改修」は本番運用の前提として保持しつつ、本学習サンプルは WP非依存の静的プロトタイプとして構築する。
- 配置先は **`site/`** ディレクトリ（テンプレートの `src/` 前提を本設計で上書き）。
- アセットは **プレースホルダで進行**（緑系ダミー画像 or CSSグラデーション＋インラインSVG図形）。実素材が来たら差し替え可能な構成にする。
- 非機能要件（モバイルLighthouse 90目標／主要テキストWCAG AAコントラスト／360px崩れ無し）を **tester の検証対象**とする。

## 3. 設計方針

**採用する方針:**
- `site/` 配下に **素のHTML/CSS/JS** でトップページ1枚（`index.html`）を構築する。ビルドツール・フレームワーク・パッケージマネージャは導入しない（ダブルクリック or ローカル静的サーバで開け、Lighthouse/ブラウザで直接検証できることを最優先）。
- カラーは **CSSカスタムプロパティ（カラートークン）** に集約し、緑系メインカラー＋単一アクセントを `:root` で一元管理する。実素材・ブランドカラー確定時にトークン値の差し替えだけで反映できるようにする。
- レイアウトは **CSS Grid / Flexbox** を用い、メディアクエリで「2カラム（PC）→ 縦積み（スマホ）」を切り替える。モバイルファースト（既定を縦積みとし、`min-width` で2カラムへ拡張）で記述する。
- ビジュアルは **「プレースホルダ写真領域 ＋ インラインSVG図形アクセント」** の二層構成。写真は緑系プレースホルダ（CSSグラデーション or ダミー画像）で代替し、芽・曲線などの図形はインラインSVGで重ねる。実写真が来たら写真領域のみ差し替える。
- 装飾は最小限（SPEC §5.4「過度な装飾はしない」）。アニメーションは `prefers-reduced-motion` を尊重し、軽量なものに限る。
- JSは最小限（モバイルメニュー開閉・CTAクリックの計測フック用 `data-*` 属性程度）。重いライブラリは持ち込まない（Lighthouse対策）。

**代替案と却下理由:**
| 案 | 却下理由 |
|---|---|
| WordPress子テーマ構成を模写 | PHP/WPランタイム無しでは描画・Lighthouse検証ができず、学習サンプルの検証性が落ちる（investigator R1/R3）。人間判断で静的方式に確定済み（U1）。 |
| React/Vue等のSPAフレームワーク | ビルド環境・依存が増え、Lighthouseのバンドルコストも上がる。1枚の静的トップに対し過剰。SPEC §6のパフォーマンス目標に逆行。 |
| CSSフレームワーク（Bootstrap等）導入 | 不要CSSの読み込みでLighthouseに不利。緑系トーン・余白設計を素のCSSで制御する方が要件（§5）に忠実。 |
| 配置先を `src/` のまま使用 | `src/` はLoop Kitフレームワーク用ツリー。人間判断で配置先 `site/` に確定済み（U3）。本設計でテンプレートの `src/` 前提を上書きする。 |

## 4. 実装詳細

> **配置先は `site/`**（テンプレートの `src/` 前提を本設計で上書き。implementerは必ず `site/` 配下に作成すること）。

**ファイル構成（追加。すべて新規）:**
```
site/
├── index.html              # トップページ本体（唯一のHTML）
├── css/
│   └── style.css           # カラートークン・レイアウト・レスポンシブを集約
├── js/
│   └── main.js             # モバイルメニュー開閉 + CTAクリック計測フック（最小限）
└── assets/                 # プレースホルダ（実素材が来たら差し替え）
    ├── logo-placeholder.svg        # ロゴのプレースホルダ（インラインSVGでも可）
    ├── mv-placeholder.（jpg/webp）  # MV写真のプレースホルダ（緑系ダミー画像）
    └── （必要に応じ favicon 等）
```
- 静的構成のため `index.html` からの相対パス（`css/style.css`・`js/main.js`・`assets/...`）で参照する。
- 実写真・実ロゴが支給され次第、`assets/` の差し替え＋必要なら `<img src>` / CSS変数の調整のみで反映できること。

**HTMLセクション構造（`<body>` 内、SPEC §5.3 の上→下の並びを厳守）:**
```
<header>            ロゴ（プレースホルダ）＋ナビ（スマホはハンバーガー）
<main>
  <section id="fv">          FV: 2カラム（左テキスト / 右ビジュアル）。2CTA並置
  <section id="business">    事業紹介: ITインフラ / SES を対等の2カラム（同一構造のカード×2）
  <section id="results">     実績・信頼（Could）: ロゴ列 or 数値 or お客様の声（プレースホルダ可）
  <section id="recruit">     採用情報（SES）: 採用ページ/専用窓口への導線プレースホルダ。親しみやすいトーン
  <section id="contact">     お問い合わせ（サービス）: CTA。フォーム本体は別途（リンク/プレースホルダ）
</main>
<footer>            会社名・コピーライト・最小限のリンク
```
- 見出し階層は `h1`（FVの大見出し、ページ内1つ）→ 各 `section` 見出しを `h2` → カード見出しを `h3` とし、論理順序を保つ（アクセシビリティ／スクリーンリーダー対応）。
- 各 `section` にはランドマーク（`aria-labelledby` で見出しと関連付け）を付与する。

**FVのマークアップ／レスポンシブ方針（SPEC §5.2）:**
```
<section id="fv">
  <div class="fv__inner">           ← CSS Grid: PC は 2カラム、スマホは 1カラム
    <div class="fv__text">          ← 左テキスト
      <h1>大見出し（プレースホルダ・キャッチコピー）</h1>
      <p class="fv__sub">サブコピー</p>
      <div class="fv__cta">         ← 2CTA並置
        <a class="btn btn--primary"   href="#contact" data-cta="contact">お問い合わせ</a>
        <a class="btn btn--secondary" href="#recruit" data-cta="recruit">採用情報</a>
      </div>
    </div>
    <div class="fv__visual">        ← 右ビジュアル（プレースホルダ写真 + インラインSVG図形）
      <picture> or <div class="fv__photo">（緑系プレースホルダ）</div>
      <svg class="fv__accent" ...>芽・曲線などの図形アクセント</svg>
    </div>
  </div>
</section>
```
- **PC（`min-width: 768px` 目安）**: `.fv__inner { display:grid; grid-template-columns: 1fr 1fr; }` で左テキスト／右ビジュアルの2カラム。
- **スマホ（既定 / `< 768px`）**: 1カラム縦積み。DOM順は「テキスト → ビジュアル → CTA」が要件（SPEC §5.2）のため、**CTAをDOM上で `.fv__visual` の後ろに置く**、または `order` でスマホ時に CTA を最後へ送る。**推奨: DOMを「テキスト（見出し+サブ）→ ビジュアル → CTA」の順に並べ**、PC時にGridの配置（`grid-template-areas` 等）でテキスト群を左・ビジュアルを右へ寄せる。これによりCSS無効時でも要件の縦積み順序が保たれる。
- ブレークポイントは360px〜の崩れ防止を最優先に設定する（実装者は360/375/768/1024で目視確認）。

**事業紹介（2事業対等）の方針（SPEC §5.3, 受け入れ条件3）:**
- ITインフラ／SES を **同一のカードコンポーネント（同じクラス・同じ内部構造）** ×2 で並べ、面積・余白・フォントサイズ・画像枠を完全に揃える。
- PCは横並び2カラム（`grid-template-columns: 1fr 1fr`、`gap` 均等）、スマホは縦積み。どちらを先に置くかで優劣を出さない（同一スタイル）。視覚的重みが等しいことをreviewer/testerが確認できるよう、寸法はCSS変数で共通化する。

**カラートークン（CSS変数 / `:root`、SPEC §5.1）:**
```css
:root{
  /* 背景: 白〜白に近い色 */
  --color-bg:        #ffffff;
  --color-surface:   #f6faf7;   /* わずかに緑寄りのオフホワイト（カード背景等） */
  /* メインカラー: 緑系 */
  --color-primary:        #1f7a4d;  /* 主要ボタン・見出しアクセント（コントラスト確保用に濃いめ） */
  --color-primary-dark:   #14603b;  /* hover/active */
  --color-primary-light:  #d8efe2;  /* 淡い緑の面・帯 */
  /* アクセント1色（CTA区別・採用セクションの差し色など。実値はブランド確定で調整） */
  --color-accent:    #e08a1e;       /* 暖色アクセント1色のみ */
  /* テキスト */
  --color-text:      #1a2b22;       /* 本文（白背景に対しAA以上） */
  --color-text-muted:#43574c;
  /* 寸法・余白トークン（2事業カードの寸法共通化にも使用） */
  --space-section:   clamp(48px, 8vw, 96px);
  --radius:          12px;
  --maxw:            1120px;
}
```
- 上記の具体値は **AAコントラストを満たす初期値の目安**。implementerはコントラスト比を確認しつつ確定し、実ブランドカラー支給時はトークン値のみ差し替える。
- 主要テキスト（本文・主要見出し・ボタン文字）はコントラスト比 ≥ 4.5:1（大文字見出しは ≥ 3:1 でも可だが原則4.5:1を狙う）を満たすこと。

**CTAのマークアップ方針:**
- CTAは `<a class="btn ...">`（アンカー）を基本とする。`.btn--primary`（お問い合わせ＝メイン緑）、`.btn--secondary`（採用情報＝アウトライン or アクセント）で色・優先度を区別。
- タップ領域は **最小44×44px**（SPEC §5.4 親指が届く大きさ）。フォーカスリング（`:focus-visible`）を明示し、キーボード操作可能にする。
- 計測は本実装スコープ外のため、`data-cta="contact"` / `data-cta="recruit"` の **data属性のみ**付与（GA4本実装は別チケット）。

**SVG図形アクセントの扱い（SPEC §5.5）:**
- 芽・曲線などの図形は **インラインSVG**（HTML内 or CSS背景）で表現し、外部画像リクエストを増やさない（Lighthouse対策）。
- 「写真＋図形アクセント」のうち写真はプレースホルダ、図形は実体としてSVGで実装する。図形は要所のみ（過剰装飾の禁止 SPEC §5.5）。装飾SVGには `aria-hidden="true"` を付与。

**JS（`main.js`）の方針:**
- ハンバーガーメニュー開閉（`aria-expanded` トグル）。
- CTAクリックの計測フック雛形（`data-cta` を読むイベントリスナの骨子のみ。実送信は別チケット）。
- 依存ライブラリなし・defer読み込み（`<script src="js/main.js" defer></script>`）。

**画像最適化方針（SPEC §6 パフォーマンス）:**
- MVプレースホルダは `webp`（または軽量JPEG）＋ `width/height` 明示（CLS防止）。
- FV直下以外の画像は `loading="lazy"`、FVのMVは `loading="eager"`/`fetchpriority="high"` 検討。
- 不要なフォント・スクリプトを読み込まない。Webフォントを使う場合は `font-display: swap`＋サブセット or システムフォントスタック優先（書体はSPEC §5.4 によりarchitect提案 → 後述）。

**フォント提案（SPEC §5.4 architectが提案）:**
- 見出し・本文ともに **モダンで可読性の高いゴシック系**を採用。第一候補は **システムフォントスタック**（`-apple-system, "Hiragino Sans", "Noto Sans JP", "Segoe UI", sans-serif`）でWebフォント0リクエストを実現し、Lighthouseと可読性を両立する。
- ブランド表現として見出しにWebフォントを使う場合は Noto Sans JP（700）等をサブセット＋`swap`で限定使用。実ロゴ受領後にロゴ書体との整合を再確認する。

**処理フロー（ページ表示時）:**
```
1. ブラウザが site/index.html を取得
2. css/style.css 読込 → カラートークン適用・モバイルファーストで縦積みレイアウト
3. min-width ブレークポイント到達時に FV/事業紹介が2カラムへ
4. インラインSVG・プレースホルダ画像を描画（写真は後日実素材に差し替え）
5. main.js（defer）でメニュー開閉と data-cta フックを初期化
```

## 6. リスク

| リスク | 深刻度 | 軽減策 |
|---|---|---|
| プレースホルダ写真がLighthouse/見た目に影響（重い・低品質） | medium | MVは軽量webp＋`width/height`明示＋`fetchpriority`。装飾はインラインSVGで画像リクエストを増やさない |
| Lighthouse Performance 90未達 | medium | フレームワーク/外部CSS不使用・Webフォント最小化・画像最適化・遅延読込。実装後に実測し未達なら最適化 |
| プレースホルダ依存で実素材差し替え時に崩れる | medium | 写真領域を固定アスペクト比（`aspect-ratio`）＋`object-fit:cover`で枠を固定し、差し替えはsrc変更のみで済む構成にする |
| 緑系×アクセントのコントラスト不足（AA未達） | medium | カラートークンの初期値をAA基準で設定し、主要テキストは実装時にコントラスト比を測定。淡色面に濃色文字を載せる |
| 360px等の狭幅でレイアウト崩れ・横スクロール | medium | モバイルファースト＋`max-width:100%`・`min-width:0`・`overflow-x`点検。360/375で実機/DevTools確認 |
| 2事業の「対等」が主観で揺れる | low | 同一カードコンポーネント＋寸法CSS変数共通化で機械的に同一寸法を保証 |
| スコープ拡大（GA4本実装・採用ページ本体に踏み込む） | low | 本チケットはFV＋2事業提示まで。計測はdata属性のみ、採用導線はリンクプレースホルダに留める |

## 9. 受け入れ条件
