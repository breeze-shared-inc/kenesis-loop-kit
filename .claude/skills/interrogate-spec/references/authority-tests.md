# authority-tests.md — 回答権限(authority)の判定手順

質問生成フェーズ(Phase 1)で、各質問の authority を `agent_recommendable` /
`human_only` のいずれかに判定するための手順。定義は question-schema.yaml を参照。

## 0. 大原則: 誤分類コストの非対称性

- human_only を agent_recommendable と誤分類すると、根拠のない一般論が推奨の体裁で
  提示され、人間に追認されるリスクがある(発見が困難な失敗)。
- 逆方向の誤分類は、対話が1往復増えるだけである(コストが低く回復可能な失敗)。
- したがって、**以下のテストで判定に迷った場合は必ず human_only に倒す**。

## 1. 判定手順(この順で適用し、最初に確定した時点で終了)

### Step 1: 検証可能性テスト

問い: 「この質問への回答の正誤を、コードベース・リポジトリ内ドキュメント・
公開情報(公式ドキュメント等)のいずれかで機械的に検証できるか?」

- **Yes** → Step 2 へ(agent_recommendable 候補)
- **No** → human_only で確定
- 判定不能・部分的にしか検証できない → human_only で確定(大原則)

### Step 2: 主語テスト

質問文を次の2形に言い換え、どちらで成立するかを確認する:

- (A) 「一般に(この技術・このドメインでは)どうすべきか?」
- (B) 「当社では(当社の規程・運用・意思決定では)どうなっているか?」

判定:
- (A) のみで成立 → **agent_recommendable** で確定
- (B) のみで成立 → **human_only** で確定
- (A)(B) 両方の要素を含む → **ハイブリッド。Step 3 へ**

### Step 3: ハイブリッド分解

ハイブリッド質問はそのまま起票せず、**必ず2問に分解する**:

1. **前半(human_only)**: 組織事実の確認。「当社の◯◯はどうなっているか」
2. **後半(agent_recommendable)**: 前半の回答を前提とした技術推奨。
   `depends_on: [前半のID]` を設定する(前半が answered になるまで提示されない)

後半の質問文には「Q-NNN の回答を前提として」と依存を明示的に書き込むこと。
investigator への調査委譲は後半についても起票時に行ってよいが、前半未回答の
時点では recommendation は条件付き(「前提が X の場合は…」)の形に留めさせる。

## 2. 分類の目安(例)

| 質問の型 | 判定 | 根拠 |
|---|---|---|
| 「Fastify でこのバリデーションはどう実装すべきか」 | agent_recommendable | 公開情報・コードベースで検証可能、一般論で成立 |
| 「freee API の勤怠タグはどの粒度で取得できるか」 | agent_recommendable | 公式ドキュメントで検証可能。※実挙動が確認できなければ調査側が surrendered で返す |
| 「月中の定期区間変更時、控除はどう按分するか」 | human_only | 真実源は当社の通勤手当規程。一般慣行は選択肢の含意としてのみ提示 |
| 「承認者が退職済みの申請は誰が承認するか」 | human_only | 組織の運用・権限設計の問題 |
| 「4.3節の承認フローは経理の月次締め実務と突き合わせ確認済みか」(fitness_concern) | human_only | 真実源は業務当事者。成果物は確認先・確認事項のリスト化(多くは T4(b) で deferred へ) |
| 「SSO はどの方式を採用すべきか」 | ハイブリッド分解 | 前半:「当社のID基盤の現状は?」(human_only)/ 後半:「その前提での推奨方式は?」(agent_recommendable, depends_on) |

## 3. authority と surrendered の関係(混同注意)

- authority は「**誰が**答えを持つか」の分類。surrendered は「**調査で**確定できたか」
  の結果である。両者は直交する。
- agent_recommendable と判定した質問でも、調査の結果確定できなければ
  surrendered: true で deferred に落ちる(例: freee API の実挙動)。
  これは判定ミスではなく、正常な流れである。
- 逆に、human_only の質問を「調査で確定できないから」と surrendered にしては
  ならない。human_only は棚上げではなく**対話で回答を得る**対象である。
  (人間が「規程を確認しないと分からない」と答えた場合に、その時点で
  deferred(T4(b))へ落とすのは正当)

## 4. 提示前の自己検査との接続

state-machine.md §5 の提示前検査は、本判定の結果がスキーマ規則
(human_only に recommendation なし、options 2件以上)に反映されているかを
機械的に確認する二重の防波堤である。判定に自信がない質問を「とりあえず
agent_recommendable にして検査で弾いてもらう」運用は禁止する。判定は生成時に行う。
