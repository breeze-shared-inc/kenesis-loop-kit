# scan-axes.md — 尋問スキャンの探索軸

Phase 1(尋問スキャン)で SPEC を読む際の探索軸。特性要因図(石川ダイアグラム)の
発想を SPEC 尋問用に翻案したもので、「思いつきの列挙」による実行ごとの質問品質の
ばらつきを抑えることを目的とする。

**使い方**: SPEC の各節に対し、以下の軸を順に当てて欠陥候補を探す。軸は「どこを
見るか」であり、発見した欠陥の分類(classification: 矛盾/未定義動作/暗黙の前提/
外部依存未知/運用適合懸念)とは独立である。1つの軸から複数の分類の質問が出てよい。

## 軸1: アクター状態

登場する人・役割の状態変化を追う。
- 退職・休職・異動した場合、その人が担っていた処理(申請・承認・管理)はどうなるか
- 権限が付与・剥奪される瞬間、進行中の処理はどちらの権限で扱われるか
- 代理・委譲・兼務は想定されているか

## 軸2: 時間境界

すべての期間・締め・タイミングの端を突く。
- 月跨ぎ・年度跨ぎ・締め日当日・締め後の遡及修正
- 有効期間を持つデータ(定期区間、規程、料金)の期間の切り替わり目
- 「即時」「月次」「翌月」などの語の正確なタイミング定義

## 軸3: データライフサイクル

データの生成前・変更中・削除後を追う。
- 参照先が先に消えた場合(マスタ削除後の履歴参照など)
- 未確定データと確定データの混在(下書き・申請中・承認済)
- 移行データ・過去データへの新規則の適用有無

## 軸4: 外部依存

外部システム・API との境界を疑う。
- 外部が落ちている/遅い/部分的に失敗した場合の挙動
- 外部の仕様変更・レスポンス形式の想定と実挙動の差(→ 多くは external_unknown)
- 二重送信・再送時の冪等性

## 軸5: 並行性・競合

同時に起きた場合を問う。
- 同一対象への同時操作(同時申請・承認とキャンセルの競合)
- バッチ処理と対話操作の重なり(月次同期中の申請変更など)

## 軸6: 権限・可視性

誰が何を見られる/できるかの境界を突く。
- 本人・上長・経理・管理者の間で、参照/編集/承認の境界は各状態で一貫しているか
- 組織変更(上長の変更)後、過去データの可視性はどうなるか

## 軸7: 運用適合(Fitness for Use)

**仕様として明確に書かれている箇所**に対し、「それは実運用に適合すると確認済みか」
を問う。ジュランの「用途への適合性」の適用であり、他の軸と性格が異なる:
テキストの欠陥ではなく、テキストと現実の突き合わせの未実施を検出する。

- この節の前提となる業務手順(経理の月次締め、承認の実務慣行、規程の運用実態)は、
  その業務の当事者と突き合わせて確認されたか
- 「システムがこう動く」と「現場がこう回っている」の間に断絶はないか

この軸から生まれる質問は classification: fitness_concern とし、原則 human_only。
成果物は回答そのものではなく**「誰に・何を確認すべきか」のリスト**である
(確認自体はセッション外の対話・プロトタイプ検証で行われるため、多くは
T4(b) で deferred に落ち、verification_hint に確認先と確認事項を記録する)。

---

## 運用注意

- 全軸 × 全節の機械的な総当たりは質問の洪水を生む。各軸を当てた上で、
  起票するかどうかは triage-rules.md §4(過剰列挙の抑制)に従うこと。
- この軸リスト自体が改善対象である。すり抜け分析(ESCAPES.md)で
  「質問が生成されなかった」に分類された欠陥は、どの軸の欠落・不足に起因するかを
  特定し、本ファイルの改訂候補とする(改訂はタンパリング防止規律に従う。README参照)。
