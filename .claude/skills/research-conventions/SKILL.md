# research-conventions — 調査規約

あなたが調査を行うとき、成果物は「答え」ではなく「根拠付きの判断材料」である。
もっともらしいが検証不能な回答は、無回答より有害である。以下の規律に常に従うこと。

## 1. ソースの種別と信頼度序列

調査で参照するソースは必ず次の5種に分類し、出力に明示する:

| kind | 内容 | 信頼度の扱い |
|---|---|---|
| `empirical` | 実測・検証結果(サンドボックス実行、動作確認、再現実験) | 最上位。実挙動そのものであり、ドキュメントと食い違う場合も実測を優先。ただし検証条件(環境・日時・手順)を summary に必ず記録 |
| `codebase` | リポジトリ内のコード・設定・マイグレーション | 高い。ただし「現状の実装」であり「あるべき仕様」ではない点を常に注記 |
| `local_docs` | リポジトリ内・ローカルのドキュメント(SPEC、設計メモ、議事録) | 高いが陳腐化しうる。最終更新時期が判別できる場合は記録 |
| `web_docs` | 公式ドキュメント・公開仕様(外部APIリファレンス等) | 公式一次情報に限り高い。ブログ・フォーラム・Q&Aサイトは補助扱いとし、単独で確信度 high の根拠にしない |
| `general_knowledge` | あなた自身の事前知識・一般的な実務慣行 | 最も低い。単独では確信度 medium が上限 |

序列は原則 empirical > codebase > local_docs > web_docs > general_knowledge。
ただし外部APIの仕様に関しては web_docs(公式)> local_docs(社内メモ)とする
(メモは伝聞・陳腐化リスク)。なお empirical は通常、調査時点では存在せず、
検証タスクの完了によって事後的に追加される種別である(調査段階で実測が必要と
判断した場合は、実測を捏造せず surrendered で返すこと。§5)。

## 2. 根拠の必須要件

- すべての調査結果に sources を付ける。各 source には kind / ref / summary を記録する。
- **ref は実在し追跡可能でなければならない**: ファイルパスは実際に開いて確認したもの、
  URLは実際に取得したもののみ。記憶から構成したパスやURLを ref に書いてはならない。
  general_knowledge の場合は「何の知識に基づくか」を1行で記述する。
- summary では**事実と推測を分離する**。「ドキュメントに明記されている」と
  「明記はないがこう解釈するのが自然」を混ぜて書かない。後者は推測と明示する。
- web_docs は取得日を summary に含める(外部仕様は改訂されるため)。

## 3. 確信度(confidence)の運用定義

- `high`: empirical・codebase または公式一次情報で直接確認でき、解釈の余地が小さい。
- `medium`: 根拠はあるが、間接的・部分的、または general_knowledge への依存が残る。
- `low`: 推測が主体、ソース間に矛盾がある、または鮮度・網羅性に重大な不安がある。

確信度は「自分の答えにどれだけ自信があるか」ではなく
「第三者が同じソースを見て同じ結論に達するか」で判定する。

## 4. 矛盾の処理(融合禁止)

- ソース間で事実が食い違う場合、**内容を融合・折衷して単一の見解を作ることを禁止する**。
- 矛盾を検出したら: (1) どのソースのどの記述が食い違うかを conflicts として報告し、
  (2) confidence を low に強制し、(3) どちらが正しいかの判定は依頼元に委ねる。
  序列上位のソースを推す場合も「序列に基づく推定」と明示し、断定しない。
- 「片方が古いだけ」と思われる場合も、その推測を理由付きで書くに留める。

## 5. 降参の出口(surrendered)

- 調査で確定できない問いに対し、**「確定不能」と報告することは正当な成果である**。
  それらしい回答の捏造は最も重大な規律違反とする。
- 次の場合は surrendered: true として報告する:
  - 実際に動かさなければ分からない(外部APIの実挙動、性能特性など)
  - 真実源が組織内の人間・規程にあり、ドキュメント化されていない
  - ソースが見つからず、general_knowledge だけでは confidence low を超えられない
- surrendered 時は必ず「何を検証・確認すれば確定するか」(verification_hint に
  相当する情報)を添える。降参は思考停止ではなく、検証タスクの設計である。

## 6. 依頼元から出力構造が指定された場合

依頼プロンプトで出力スキーマ(例: interrogate-spec の research 構造)が指定された
場合はそれに厳密に従う。特に:

- **authority = human_only と指定された問いには recommendation を生成しない**。
  選択肢(options)と各選択肢の含意(implications)の列挙に徹する。
  「一般的にはこうする」という知識は、推奨ではなく選択肢の含意の中で述べる。
- alternatives には推奨案との得失(tradeoff)を必ず付け、単なる列挙にしない。

## 7. 調査範囲の自制

- 依頼された問いに答えるための最小限の調査に留める。関連する興味深い発見があれば
  出力末尾に1〜2行で言及してよいが、深掘りは依頼元の判断に委ねる。
- 同一ファイル・同一ページの再取得を避け、調査ログを不必要に肥大化させない。
