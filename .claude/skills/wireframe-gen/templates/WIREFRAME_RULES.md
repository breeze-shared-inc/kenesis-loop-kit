# WIREFRAME_RULES.md — ワイヤーフレームHTML生成規約

> このファイルはワイヤーフレーム生成Skillが従うマスター規約です。
> 生成されるすべてのHTMLファイルはこの規約に準拠すること。

## 1. ファイル規約

- **形式**: 単一ファイルの静的HTML。1画面 = 1ファイル。
- **配置**: `docs/wireframes/` 直下。サブディレクトリは作らない。
- **命名**: `SCR-ID-画面名英語.html`(例: `SCR-001-login.html`)。英語名は小文字ケバブケース。
- **ハブページ**: `docs/wireframes/index.html`(命名固定)。
- **文字コード**: UTF-8。`<meta charset="utf-8">` を必ず含める。
- **外部依存ゼロ**: CDN・外部CSS/JS・画像ファイル・Webフォントの参照を禁止する。CSSは`<style>`タグ内、アイコンや図はインラインSVGまたはUnicode文字で表現する。理由: エージェントがファイル単体で全情報を読めること、オフラインでも表示が崩れないことを保証するため。
- **JavaScript**: 原則使用しない。画面内の表示確認に不可欠な場合(タブ切替・モーダル開閉など)のみ、`<script>`タグ内に最小限記述してよい。フォーム送信・API呼び出し・状態管理のコードは書かない。

## 2. 冒頭コメント(必須)

すべてのワイヤーフレームHTMLの1行目(DOCTYPE直後)に、以下のコメントを挿入すること:

```html
<!--
  ⚠ このファイルはワイヤーフレーム(画面イメージの確認用)です。実装コードではありません。
  実装は docs/SPEC.md セクション9(技術的制約)に定められた技術スタックで別途行うこと。
  このHTML/CSSを本番コードへコピーしてはならない。
  対応画面: {SCR-ID} {画面名} / 準拠SPEC: v{X.X} / 生成日: {YYYY-MM-DD}
-->
```

## 3. 画面情報バー(必須)

各画面の最上部に、レビュー時の識別用として次の情報バーを固定表示すること(スクリーンショット共有時にも画面を特定できるようにするため):

- 表示内容: `{SCR-ID} {画面名} | ワイヤーフレーム | 準拠SPEC v{X.X}`
- スタイル: 画面幅いっぱいの細い帯(高さ32px程度・濃いグレー背景・白文字・小さめフォント)。本体デザインと混同されない見た目にする。
- バーの右端に `index.html` へ戻るリンクを置く。

## 4. 忠実度(中忠実度)の基準

本プロジェクトのワイヤーフレームは**中忠実度**とする。完成イメージの共有が目的である。

- **配色**: プロジェクトで指定されたカラーパレット(後述5)を使用する。指定がない場合はデフォルトパレットを使う。
- **寸法感**: 実際の利用を想定したコンテナ幅・フォーム幅・ボタンサイズにする(例: PCフォームは最大幅480px程度、タップ対象は最小44px)。
- **ダミーデータ**: 現実的な内容にする(「山田太郎」「suzuki@example.com」「2026-07-04」等)。「テキストテキスト」「aaa」のような無意味な埋め草は使わない。
- **作り込まないもの**: アニメーション、ホバー時の凝った演出、faviconなど。目的は完成イメージの共有であり、ピクセル単位の再現ではない。

## 5. カラーパレット

- **決定方法**: 初回生成時に一度だけユーザーへ「ブランドカラーや参考にしたい既存サイト・資料はありますか?」と確認する。指定があればそのブランドカラーイメージでパレットを組み、指定がなければ以下のデフォルトパレットを使う。
- **デフォルトパレット**(使用比率の目安付き):

| 変数名 | 色 | 役割 | 使用比率の目安 |
|---|---|---|---|
| --color-base | #FFFFFF | 背景・カード面 | 65〜70%(画面の大部分は白ベースを保つ) |
| --color-main | #64B9E6 (sky) | 罫線的な装飾線・アイコン背景・見出し帯（**非テキスト装飾に限定**。白背景でのコントラストが約2.2:1しかないため、文字色・リンク色には使わない） | 18〜22% |
| --color-main-deep | #2E8BC9 (sky_deep) | 塗りボタン・選択状態・バッジ・**リンク色**(**文字は白のみ**。この色の上に黒系文字を置かない) | mainの内数 |
| --color-link | #1C6FA6 | 白背景上のテキストリンク（sky系でコントラスト4.5:1以上を確保した濃色） | — |
| --color-accent | #FF4568 (pink) | 強調ラベル・必須マーク・箇条書きマーカー・エラー表示 | 8%以下 |
| --color-accent-2 | #F9BA5D (orange) | 写真/画像プレースホルダ背景・アイコン背景の変化付け | 5%以下 |
| --color-text | #374151 | 本文文字(**純黒 #000000 は使わない**) | — |

> **コントラスト規律（§10と整合）**: 白背景上に置く文字・リンク・小さなUIラベルは、AA基準（本文4.5:1・大きめの文字/UI部品3:1）を満たす色に限る。sky (`--color-main`) は 2.2:1 のため**テキスト用途で使わない**。白背景のテキストリンクは `--color-link`、塗りボタン（白文字）は `--color-main-deep` を使う。ブランド指定パレットを組む場合も同じ下限を守ること。

- **派生色**(デフォルトパレット使用時のみ許可する補助色。これ以外の色を追加しない):
  - --color-border: #D6E9F5(skyの淡色。カード枠・入力欄枠・区切り線)
  - --color-bg-subtle: #F0F7FC(skyの極淡色。テーブルの縞・非活性領域の背景)
  - --color-text-sub: #6B7280(補助文字・プレースホルダ)
- **アクセント色の規律**: pinkとorangeは面積の小さい要素に限定する。ボタンの基本色はmain-deepであり、pinkの塗りボタンは「削除」などの破壊的操作か、エラー状態の表示にのみ使う。
- **記録場所**: 決定したパレット(ブランド指定・デフォルトのいずれでも)は `index.html` 内のコメントに記録し、全画面で統一する。**SPEC.mdには書かない**(見た目の正はワイヤーフレーム側、という責務分担を守るため)。
- CSSでは上記をCSSカスタムプロパティとして`:root`に定義し、直値の散在を避ける。

## 6. レスポンシブ(モバイルレイアウト必須)

すべての画面ワイヤーフレームは、PCレイアウトとモバイルレイアウトの**両方を1ファイル内に定義**する。可変レイアウト任せにせず、メディアクエリで明示的にモバイル時の変形を書くこと(生成のたびにモバイル対応の有無が変わる、という再現性のなさを防ぐため)。

- `<meta name="viewport" content="width=device-width, initial-scale=1">` を必ず含める。
- **ブレークポイント**: 640px以下をモバイルとする(`@media (max-width: 640px)`)。モバイルの設計基準幅は375px。
- **標準変形ルール**(モバイル時):
  - 複数カラムは縦積みにする(サイドバーは本文の下または折りたたみ)
  - グローバルナビゲーションはハンバーガーアイコン表示にする(ワイヤーフレームでは展開後のメニューを静的に併記するか、注記で示せばよい。JSでの開閉実装は不要)
  - 主要ボタンは全幅にする。タップ対象は最小44pxを維持する
  - テーブルは横スクロール(`overflow-x: auto`)またはカード型への変形とし、どちらを採るか画面ごとに選ぶ
  - 余白・フォントサイズはモバイル向けに詰めるが、可読性(本文14px以上)を維持する
- モバイル時に**表示を省略する要素**がある場合は、勝手に消さず、画面内の注記または生成後の報告で明示する(モバイルでの要素の取捨選択は画面仕様の一部であり、暗黙に決めてはならない)。

## 7. 画面間リンク(遷移の再現)

- SPECセクション6の遷移図に定義された遷移は、対応するボタン・リンクに実ファイルへの相対リンク(`<a href="SCR-002-dashboard.html">`)を張り、ブラウザ上で遷移を辿れるようにする。
- 条件分岐する遷移(ログイン成功/失敗など)は、成功側をリンクにし、失敗側は画面内にエラー表示例を静的に併記するか、注記で示す。
- 未作成の画面への遷移は、リンクにせずグレー表示+「(未作成: SCR-xxx)」の注記を付ける。

## 8. ダミーデータ・記載禁止事項

- 実在の個人名・メールアドレス・電話番号・住所を使わない(example.com / 架空の氏名を使う)。
- APIキー・パスワード等のシークレットの実値を、例としても記載しない。
- 実在企業のロゴ・ブランド名を使わない(自プロジェクト名は可)。

## 9. index.html(ハブページ)と compare.html(比較ビュー)の構成

### index.html

1. **タイトルとメタ情報**: プロジェクト名、準拠SPEC版数、最終更新日。
2. **画面一覧**: SPECセクション6と同じ順で、カード形式(SCR-ID・画面名・目的・対応要件ID・ステータス[作成済/未作成]・画面を開くリンク・**「PC/スマホ比較」リンク**)。
3. **画面遷移図**: SPECのMermaid遷移図を、外部依存ゼロを守るため**手描きのインラインSVGまたはCSSレイアウト**で再現する(Mermaid.jsは読み込まない)。各ノードは対応するHTMLへのリンクにする。
   - **表現方法の選択（座標破綻を避ける）**: 画面数が少なく分岐も単純（目安: ノード7個以下・一直線〜軽い分岐）ならインラインSVGで座標を置いてよい。それを超える、または双方向・合流の多い複雑な遷移では、**座標計算の要らないCSSレイアウト**（ノード=カードを縦/横に並べ、遷移を「→ ラベル」付きのリンク行や矢印記号で表す）にフォールバックしてよい。無理に大きなSVGを手座標で描いてノードの重なり・矢印のずれを出すより、崩れない表現を優先する。
   - どちらの表現でも、遷移のラベル（「ログイン成功」等）と方向がSPECのMermaid定義と一致していること。未作成画面のノードはグレー表示にする。
4. **レビュー手順の説明**: 非エンジニア向けに「カードをクリックして画面を開く」「『PC/スマホ比較』でモバイル表示を並べて確認できる」「修正してほしい点はSCR-IDと場所を添えて伝える」を短く記載する。
5. 画面の追加・削除・改名があった場合、index.htmlは必ず再生成する(手動編集で差分対応しない)。

### compare.html(PC/スマホ並列プレビュー)

画面ごとの比較ページを量産する代わりに、`docs/wireframes/compare.html` を1ファイルだけ生成する:

- URLクエリ(例: `compare.html?screen=SCR-001-login.html`)で対象画面を受け取り、同じHTMLを**幅の異なる2つのiframe**(PC: 幅100%・スマホ: 幅375px×高さ667pxの端末風フレーム)で並べて表示する。iframeの幅がビューポート幅として働くため、メディアクエリによるモバイルレイアウトがそのまま確認できる。
- クエリなしで開かれた場合は、画面一覧へのリンクを表示する。
- **JS使用の例外**: index.html と compare.html に限り、レビュー補助(クエリの読み取り・iframe切替など)のための最小限のJSを許可する。画面ワイヤーフレーム本体のJS原則(セクション1)は変わらない。外部依存ゼロの原則はこの2ファイルにも適用する。

## 10. アクセシビリティ(最低限)

- フォーム部品には`<label>`を対応付ける。
- 見出しは`<h1>`から順に階層を守る。
- 文字色と背景のコントラストは、補助文字を含めて読める水準を保つ。

これらは実装時の品質要件の先取りではなく、レビュー時に「読める・操作対象がわかる」状態を保つための最低限である。
