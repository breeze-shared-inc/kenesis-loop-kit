# SPECインタビュー — 対話形式での要件定義書作成

## 目的と成果物

非エンジニアのユーザーと対話し、kenesis-loop-kit の開発ループの前提となる要件定義書 `docs/SPEC.md` を作成する。

- **成果物**: `docs/SPEC.md`(`templates/SPEC_TEMPLATE.md` の構成に完全準拠したもの)
- **成功条件**: 後述の「完成チェックリスト」をすべて満たし、ユーザーが内容を承認していること

## 参照ファイル

| ファイル | 読むタイミング |
|---|---|
| `templates/SPEC_TEMPLATE.md` | インタビュー開始前に必ず全体を読む。生成するSPECの構造・ID採番規則・各セクションの記述規約はすべてこのファイルが正 |
| `templates/DEFAULT_STACK.md` | セクション9(技術的制約)のヒアリングに入る前に必ず読む。デフォルト適用の条件(反証質問)と転記ルールが定義されている |

## 対話の基本原則

このスキルの利用者は要件定義の経験がない非エンジニアであることを前提とする。以下を守らないと、ユーザーが答えられずにインタビューが破綻するか、曖昧な回答がそのままSPECに入り込み、後続の自律開発ループでブロッカーになる。

1. **専門用語を使わずに聞く。** 「機能要件」「非機能要件」「ステークホルダー」といった言葉を質問文に使わない。「絶対に必要なことはどれですか?」「誰が使いますか?」「同時に何人くらい使いそうですか?」のように、日常の言葉で聞く。
2. **一度に聞くのは1〜2問まで。** セクションの項目を一括で並べて聞かない。回答を受けて次の質問を決める。
3. **要件記法への翻訳はこちらの仕事。** ユーザーの発話(「ログインは必要」「スマホでも見たい」)を、REQ-ID付きの「〜できること」形式や非機能要件の表形式に変換するのはこのスキルの責務であり、ユーザーに書式を要求してはならない。翻訳結果はそのセクションのヒアリング終了時にユーザーへ見せ、認識齟齬がないか確認する。
4. **「わからない」「まだ決めていない」は正当な回答として扱う。** 無理に答えを引き出さず、その場でOQ-IDを振って未決事項(セクション12)に登録する。「後で決められます。いったん宿題リストに入れておきますね」と伝えて先へ進む。
5. **回答から推測で補完しない。** ユーザーが言っていないことを要件に昇格させない。推測が必要な場面では「〜という理解で合っていますか?」と確認質問に変換する。
6. **具体例で聞く。** 抽象的な質問(「セキュリティ要件はありますか?」)ではなく、状況で聞く(「このシステムには他人に見られると困る情報が入りますか?例えば住所や給与などです」)。

## 進め方

### Step 0: モード選択

`/spec-interview` に引数が渡されている場合($ARGUMENTS が「最短」または「フル」)、モードの質問を省略してそのモードで開始する。引数がない場合・発話から自動起動した場合は、所要時間の目安とともに2つの進め方を提示し、ユーザーに選ばせる。

- **最短コース(約10〜15分)**: セクション1(概要)・2(背景)・4(誰が使うか)・5(機能要件のMustのみ)・6(画面一覧)・11(スコープ外)だけを聞く。セクション4を聞くのは、セクション6の「アクセス可能なユーザー」列を確定するのに最低限のユーザー種別が要るため(「使う人は1種類だけですか?管理する人と一般の利用者は分かれますか?」の1問でよい)。残りのセクションはテンプレートの規約に従い、デフォルト適用(技術的制約)・OQ登録(未決扱い)・標準値の提案(非機能要件)で埋める。まず動く要件定義を素早く作りたい場合向け。
- **フルコース(約30〜45分)**: 全セクションを順に聞く。

どちらのモードでも、生成されるSPEC.mdの構造は同一とする(セクションを省略しない)。差は「人間に聞くか、デフォルト/OQで埋めるか」だけである。

### Step 1: インタビュー(セクション順)

テンプレートのセクション順にヒアリングする。各セクションでの聞き方の指針:

- **1. プロジェクト概要 / 2. 背景・目的**: 「どんなものを作りたいですか?」「今、何に困っていますか?それがどうなったら成功ですか?」。背景で得た回答は後の成功基準(セクション13)の材料になるため、定量的な言及(時間・件数・金額)が出たら覚えておく。
- **3. 用語集**: 独立した質問はしない。会話中にユーザーが使った業務用語・多義的な言葉(「ユーザー」「申請」「承認」など)を拾い、意味を確認して登録する。
- **4. ユーザー・ステークホルダー**: 「誰が使いますか?」「管理する人と一般の利用者は分かれますか?」
- **5. 機能要件**: 「絶対に必要なこと」「あったら嬉しいこと」「余裕があればやりたいこと」の3段階で聞き、Must/Should/Couldに振り分ける。Must項目には必ず「それがうまくいかなかったとき、どうなってほしいですか?」と失敗時の挙動を聞く(例: ログインに失敗したら?)。
- **6. 画面一覧・画面遷移**: 「利用者が見る画面を、思いつく順に挙げてください」から始め、機能要件との対応を取りながらSCR-IDを振る。ユーザーが画面を挙げきれない場合は、機能要件から必要と思われる画面を**提案**し、承認を得て登録する(勝手に確定しない)。遷移はMermaid図に変換して見せ、「この流れで合っていますか?」と確認する。ワイヤーフレームのパス欄はこの時点ではすべて「(未作成)」とする。
- **7. エラーハンドリング / 8. 非機能要件**: 非エンジニアが最も答えにくい領域。直接聞かず、状況で聞く: 「何人くらいが同時に使いそうですか?」「止まると一番困るのはいつですか?」「他人に見られると困る情報は入りますか?」。回答を表形式の要件に翻訳し、検証方法の列はこちらで埋める(自動検証できないものは「人間が確認」と明記)。回答が得られない項目はテンプレートの例に準じた標準値を**提案**し、承認を得て採用するか、OQ化する。
- **9. 技術的制約**: 後述「技術スタックの扱い」に従う。
- **10. データ・外部インターフェース**: 「このシステムに入れる情報の中で、特に大事・秘密にすべきものはどれですか?」「他のサービス(メール送信、決済など)と連携しますか?」。シークレットの値は絶対にSPECに書かず、環境変数名のみ記載する。
- **11. スコープ外**: 「今回は**やらない**と決めていることはありますか?」に加え、会話中に出た「いつかやりたい」事項をこちらから「今回はやらない、に入れますか?」と確認して登録する。スコープ外が空のSPECは自律ループで過剰実装を招くため、必ず最低1項目は確定させ(完成チェックリストのゲート)、フルコースでは2項目以上を目標に引き出すよう努める。
- **12. 未決事項**: 独立した質問はしない。インタビュー全体を通してOQ化した項目がここに集約される。最後に一覧を読み上げ、「この宿題リストで漏れはないですか?」と確認する。
- **13. 成功基準**: 背景・目的で得た定量的な言及と、Must要件・非機能要件のIDを組み合わせてこちらで草案を作り、承認を得る。

### Step 2: 技術スタックの扱い(セクション9)

`templates/DEFAULT_STACK.md` を読み、その定義に厳密に従う。要点:

1. まずユーザーに技術の指定・希望があるか一度だけ聞く(「使う技術やサービスについて、会社やエンジニアからの指定はありますか?」)。指定があれば★付きで採用する。
2. 指定がなければ、DEFAULT_STACK.md セクション2の**反証質問5問**を順に行う。これは技術の質問ではなく、非エンジニアが答えられるビジネス上の事実確認である。
3. 反証に該当しなければ標準構成をSPECセクション9へ**実体転記**する(参照書き禁止)。出自注記・★・OQ化の書式はDEFAULT_STACK.md セクション3の転記ルールに従う。
4. ユーザーには「技術的な部分は、当社の標準構成で仮置きしました。エンジニアの確認で変更される可能性があります」と平易に伝える。

### Step 3: 生成と完成チェックリスト

SPEC.md全文を生成する前に、以下をセルフチェックする。1つでも満たさない場合は生成せず、不足箇所のヒアリングに戻るか、OQ化して埋める。

- [ ] すべての機能要件にREQ-IDが振られ、Must要件はすべて失敗時の挙動を持つ
- [ ] Mustに、未決事項(OQ)に依存する要件が含まれていない(依存するものはShould以下へ降格し、備考にOQ-IDを記載済み)
- [ ] すべての非機能要件に検証方法が記載されている
- [ ] 画面一覧の全SCR-IDが機能要件のREQ-IDと相互参照されている(対応画面のない要件は「-」と明示)
- [ ] 画面一覧の各SCRの「アクセス可能なユーザー」がセクション4のユーザー種別のいずれかに対応している(最短コースでも本項は満たすこと)
- [ ] UIを持たないプロジェクトの場合、セクション6に「該当なし」と明記されている
- [ ] セクション9に出自注記(標準構成の版数 or ユーザー指定★)がある
- [ ] スコープ外に最低1項目ある(フルコースでは2項目以上を目標に引き出す)
- [ ] シークレットの実値がどこにも書かれていない(環境変数名のみ)
- [ ] 変更履歴(セクション0)にv1.0の行があり、日付が今日になっている
- [ ] テンプレートの全セクション(0〜14)が存在し、削除されたセクションがない

チェック通過後、SPEC.md全文を生成してユーザーに提示し、修正希望を反映してから `docs/SPEC.md` に保存する。

### Step 4: 保存と次工程への引き継ぎ

1. `docs/SPEC.md` として保存する(既存ファイルがある場合は後述「改訂モード」に従う)。
2. 保存後、次のように案内する: 「SPECができました。次のステップは各画面のワイヤーフレーム(画面の下書き)作成です。続けて進めますか?」— ワイヤーフレーム生成は**別スキルの責務**なので、このスキル内では作成しない。ユーザーが希望したらワイヤーフレーム生成スキルの利用に引き継ぐ。
3. OQが残っている場合は、「エンジニアに確認が必要な宿題が◯件あります」とOQ一覧を提示し、確認先(社内エンジニア・ベンダー等)を決めるよう促す。

## 改訂モード(既存SPECがある場合)

`docs/SPEC.md` が既に存在する状態でこのスキルが起動された場合:

1. 既存SPECを読み、変更したい箇所をユーザーに聞く(ゼロからのインタビューをやり直さない)。
2. 変更はテンプレートの規約に従う: ID(REQ/SCR/NFR/OQ)は欠番が出ても再利用しない。決定済みOQはステータスを「決定済」にし、決定内容を該当セクションへ反映する。
3. 必ずセクション0の変更履歴に行を追加し、版数を上げる。
4. 画面一覧を変更した場合は、「対応するワイヤーフレームの更新が必要です」と明示的に伝える(SPECとワイヤーフレームの乖離を放置しない)。

## してはならないこと

- ユーザーの回答がないまま要件を確定すること(提案と確定を区別し、確定には必ず承認を得る)
- テンプレートの構成・ID規則を独自に変更すること(構造の正はあくまで SPEC_TEMPLATE.md)
- 技術選定についてユーザーと技術論を始めること(指定の有無と反証質問の確認のみ行い、選定判断はデフォルト構成またはエンジニアに委ねる)
- ワイヤーフレームや実装コードの作成に踏み込むこと(それぞれ別スキル・開発ループの責務)
