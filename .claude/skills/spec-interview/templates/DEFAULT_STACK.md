## 1. 標準構成

### 1.1 言語・ランタイム

| 項目 | 指定 | 備考 |
|---|---|---|
| 使用言語 | TypeScript 5.x | フロントエンド・API共通。`strict: true` を必須とする |
| ランタイム | Node.js 22 (LTS) | LTS版を使用。マイナーバージョンはDockerイメージで固定する |
| パッケージマネージャ | npm | lockファイル(package-lock.json)を必ずコミットする |

### 1.2 アプリケーション

| 項目 | 指定 | 備考 |
|---|---|---|
| APIフレームワーク | Fastify 5 | REST APIとして構築。スキーマ定義にはTypeBoxを使用する |
| フロントエンド | React 19 | ビルドツールはVite。SPAとして構築する |
| CSS | TailwindCSS 4 | レスポンシブ対応必須(モバイル・タブレット・デスクトップ)。UIコンポーネントライブラリの追加導入は要件に応じて可 |
| テスト | Vitest | 単体テストに使用。カバレッジ計測もVitestで行う(kenesis-loop-kitのQuality Gate対応) |

### 1.3 データベース

| 項目 | 指定 | 備考 |
|---|---|---|
| データベース | PostgreSQL 17 | Dockerコンテナで稼働。バージョンはイメージタグで固定する |
| マイグレーション | 採用ORMに付属するマイグレーション機能を使用 | ORMの選定(Prisma / Drizzle等)はarchitectの設計判断に委ねる |

### 1.4 システム基盤

| 項目 | 指定 | 備考 |
|---|---|---|
| 基盤 | VPS + Docker | docker compose によるコンテナ構成 |
| コンテナ分割 | api / frontend / db / proxy の4コンテナを基本とする | frontendはビルド成果物をproxyから静的配信する構成も可(architectの設計判断) |
| リバースプロキシ・SSL | Caddy または nginx + Let's Encrypt | HTTPS必須 |
| シークレット管理 | 環境変数(.envファイル) | .envはGit管理外とし、SPECおよび設計書には環境変数名のみを記載する(CLAUDE.mdセキュリティ規約に準拠) |

## 2. 適用条件(反証質問チェックリスト)

本構成は「無条件適用」ではなく「**反証がなければ適用**」とする。
SPECインタビューSkillは適用前に以下を必ず確認し、1つでも該当した項目はデフォルトを適用せず、OQ(未決事項)として「エンジニアへの確認事項」に登録すること。

| # | 確認質問(非エンジニア向けの聞き方) | 該当した場合に崩れる前提 |
|---|---|---|
| 1 | 既にある社内システムやデータベースに追加する話ですか?それとも完全に新規ですか? | 既存資産があるなら DB・言語・基盤の前提が崩れる → 該当項目をOQ化 |
| 2 | 会社としてサーバーやクラウドサービスの指定はありますか? | 指定があるなら「VPS + Docker」が崩れる → 基盤をOQ化 |
| 3 | サーバーの面倒(障害対応・アップデート・バックアップ)を見る担当者やベンダーはいますか? | いないなら VPS運用が成立しない → 「運用体制未定」をOQ化(基盤選定の再検討事項として記録) |
| 4 | スマートフォンの「アプリ」(ストアから入れるもの)は必要ですか? | 必要なら Webスタック自体の前提が崩れる → フロントエンド方針をOQ化 |
| 5 | 使う人数の規模感はどれくらいですか?(数人〜数百人/数千人以上/不明) | 数千人以上または不明の場合、単一VPS構成の妥当性が崩れる → 基盤・可用性をOQ化 |

## 3. SPEC.mdへの転記ルール

SPECインタビューSkillが本構成を適用する際は、以下に従うこと。

1. 本ファイルのセクション1の内容を、SPEC.mdのセクション9(技術的制約)へ実体として転記する(「標準構成に従う」等の参照書きは禁止)。
2. セクション9の冒頭に出自の注記を自動挿入する。書式は次のとおり:

   > 本構成は標準構成 v1.0 (YYYY-MM-DD) をデフォルト適用したものである。
   > ユーザーによる明示指定項目には ★ を付す。

3. ユーザーが明示的に指定した項目(例: 「データベースは会社指定のMySQLを使う」)は、デフォルトより優先して転記し、行頭に ★ を付す。
4. 適用条件チェックでOQ化した項目は、セクション9には「未決定(OQ-xxx参照)」と記載し、SPEC.mdのセクション12(未決事項)にエンジニアへの確認事項として登録する。
5. 転記後、本構成の適用可否に影響した回答(既存システムの有無・運用担当の有無など)は、SPEC.mdのセクション2(背景)またはセクション9の備考に事実として残すこと。

## 4. 本構成が前提とするプロジェクト像(参考)

- 新規開発のWebアプリケーション(SPA + REST API)
- 利用規模: 数人〜数百人程度の社内・小規模サービス
- 運用: VPSの保守を担えるエンジニアまたはベンダーが存在する
- UIあり(UIを持たないAPI・バッチのみのプロジェクトでは、フロントエンド関連項目を「該当なし」として転記する)

この像から大きく外れるプロジェクトでは、本構成を機械的に適用せず、該当項目をOQ化してエンジニアの判断を仰ぐこと。
