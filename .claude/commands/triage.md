# /triage

CLAUDE.md「長期放置チケットのポリシー」に従い、`blocked` のまま放置されたチケットをトリアージしてください。ループを起動せずにこのコマンド単体で実行できます（/start-loop の起動時チェックリストからも参照されます）。

## 引数

- 形式: `[日数 | チケットID]`（省略可）
- 省略時: `updated` から**14日以上**経過した `blocked` チケットを対象とする
- 数値を指定: しきい値の日数を変更する（例: `/triage 7`。`0` で全blockedを対象）
- チケットIDを指定: 経過日数に関わらずそのチケットをトリアージする（例: `/triage APP-003`）

## 手順

1. tickets/active/ の全チケットを読み取り、`status = blocked` かつ条件（経過日数またはID一致）を満たすチケットを一覧表示する（ID・タイトル・ブロッカー内容・`updated` からの経過日数）
2. 対象が0件なら「トリアージ対象はありません」と報告して終了する
3. 対象チケットを1件ずつ人間に提示し、「再開 / クローズ / 期限延長」のいずれかを確認する。判断に必要な情報（ブロッカーセクション・直近のログ）を添える
4. 人間の判断に応じて以下を実行する。チケットの変更は必ずWrite/Editツールで行う（Bashリダイレクト・sed・tee禁止）

   **再開**（ブロッカーが解消された）
   - 元のstatusを特定する: `tickets/.metrics.jsonl` から該当チケットの `"to": "blocked"` である最後のイベントを探し、その `from` を元statusとする。イベントが無い場合はチケットのログセクションから推定し、それでも特定できなければ人間に確認する
   - statusを元のstatusへ戻す
   - ブロッカーセクションの解消済み記載を削除する（チケットテンプレートの「解消後削除」）
   - ログに「YYYY-MM-DD HH:MM: トリアージ: 再開 - {元status}へ復帰」を追記し、`updated` を現在日時に更新する
   - 再開後にループを回す場合は `/start-loop {チケットID}` を案内する

   **クローズ**（不要になった）
   - statusを `cancelled` に変更する（tickets/active/ に置いたまま変更する。PreToolUse検証とメトリクス記録を効かせるため）
   - ログに「YYYY-MM-DD HH:MM: トリアージ: クローズ - {理由}」を追記し、`updated` を現在日時に更新する
   - その後、tickets/done/ へBashで移動して保管する

   **期限延長**（引き続き対応待ち）
   - ブロッカーセクションに対応期限を追記する（例: 「期限: YYYY-MM-DD（{理由}）」）
   - ログに「YYYY-MM-DD HH:MM: トリアージ: 期限延長 - {期限}」を追記し、`updated` を現在日時に更新する（14日タイマーがリセットされる）

5. 全件の処理結果（再開/クローズ/期限延長/スキップの内訳）を一覧で報告する

## Never

- 人間の判断なしにstatusを変更しない（トリアージの三択は人間が決める）
- ログセクションの既存記載を削除・上書きしない（ブロッカーセクションの解消済み記載の削除は除く）
- `cancelled` にしたチケットを再びループの処理対象にしない

## 使用例

```
/triage
/triage 7
/triage APP-003
```
