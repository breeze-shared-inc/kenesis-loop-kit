# /setup

Kenesis Loop Kitを導入したプロジェクトの初期セットアップと現在地診断を実行してください。

**全ステップは冪等です。** 各ステップは「状態を確認し、済みなら結果を報告してスキップ」で統一します。初回導入時だけでなく、途中まで進んだプロジェクトで再実行しても常に正しい「次の一手」を案内します。

## 手順

### 1. リポジトリの初期化判定（Kitからの切り離し）

`git clone` でKitを導入した場合、新プロジェクトの `origin` がKit本体（`kenesis-loop-kit`）を指したままになる。誤pushやKit履歴の混在を防ぐため、この状態を検出して**クリーンな履歴での再初期化**を案内する（GitHubの「Use this template」経由で導入した場合は既に切り離されているため、この手順は自動でスキップされる）。

1. `git config --get remote.origin.url` を確認する
2. URLが `kenesis-loop-kit` を**含まない**（または `origin` が未設定）→ 既に切り離し済み。結果を報告してスキップする（冪等）
3. URLが `kenesis-loop-kit` を**含む**場合、切り離しの前に人間へ必ず次を確認する
   - **これはKit本体そのものの開発か**（このリポジトリでKitを改良している等）→ **はい なら何もしない**。誤爆防止のためこの確認を省略しない
   - Kitをテンプレートとして新プロジェクトを始める段階か → 次へ
4. 切り離しに同意が得られた場合**のみ**、以下を実行する。**`.git` の削除は不可逆**であり、実行前に必ず人間の明示承認を取る。承認が無い・不明な場合は実行しない（安全側）

   ```bash
   rm -rf .git && git init
   ```

   - これによりKitのコミット履歴は引き継がれず、クリーンな履歴で新プロジェクトを開始する。**Kitのその後の更新は取り込めなくなる**点も併せて伝える
   - 実行後は `origin` が無い状態になる。自分のリポジトリを用意したら `git remote add origin <URL>` で接続するよう案内する（GUI/認証を伴うためこのコマンドからは実行しない）

### 2. .gitignoreプリセットの選択

1. 現在の `.gitignore` を `.gitignore.public` / `.gitignore.private` と比較する
   - **どちらとも一致しない場合**（独自カスタマイズの可能性）は差分を提示し、上書きしてよいか人間に確認する。確認なしに上書きしない
2. 人間に「プライベートリポジトリで実チケット（tickets/active・done）もコミットしたいか」を確認する
   - はい → `cp .gitignore.private .gitignore`
   - いいえ / 不明 → 変更しない（デフォルトのpublicプリセットは実チケットを除外する安全側）
3. プリセットの中身の前提: publicプリセットが除外するのはチケット本体（tickets/active/*.md・tickets/done/*.md）のみ。tickets/Templates/ と tickets/_index.md はどちらのプリセットでも追跡対象として配布される（パブリックリポジトリのclone直後もこれらのファイルは存在する前提でよい）

### 3. Obsidianの案内（任意）

- Obsidianは必須ではないことを伝える（実体はMarkdownファイル群であり、エディタだけで全機能が動作する）
- 使う場合の手順（`tickets/` をvaultとして開く・推奨プラグイン）は `docs/obsidian-setup.md` を案内する。GUI操作のためこのコマンドからは実行しない

### 4. プロジェクト略称の確定

1. CLAUDE.md「チケットID採番」節に「このプロジェクトの略称」の行があるか確認する
2. 既にあれば現在の略称を報告してスキップする
3. 無ければ人間にチケットIDの略称（例: `APP`）を確認し、同節へ以下の形式でEditツールで追記する（`/new-ticket` が採番と `project` フロントマターに使用する）

   ```
   - このプロジェクトの略称: `XXX`
   ```

### 5. SPEC・前工程の現在地診断

以下を上から順に判定し、**最初に該当したもの**を「次の一手」として案内する。このコマンドではSPEC.mdやチケットを作成せず、必ず専用のスキル/コマンドへ誘導する。

SPECが存在する場合は、判定の前に構造チェックを実行する:

```bash
python3 .claude/skills/spec-interview/scripts/check_spec_structure.py docs/SPEC.md
```

FAILが1件以上あれば「構造チェックNG（テンプレート非準拠）」と判定する（WARNのみは準拠として扱い、内容を結果報告に含める）。外部で作成されたSPECを持ち込んだ場合はテンプレート非準拠のことが多く、非準拠のまま先へ進むと `/wireframe-gen` の受付やチケット起票のトレーサビリティ（REQ/SCR/IF-ID）で差し戻されるため、先に構造化する。構造チェックNGかつ `docs/spec-qa/<spec名>/` が既に存在する（＝尋問開始済み）場合は、構造化によりSPEC改訂検知の再スキャンと質問アンカー（spec_refs）切れが発生する旨を併せて伝える。

| 状態 | 次の一手 |
|---|---|
| `docs/SPEC.md` が無い | `/spec-interview` で要件定義を作成する |
| SPEC有り・構造チェックNG（テンプレート非準拠） | `/spec-interview` の改訂モードでテンプレート準拠に構造化する（成果物=SPEC_TEMPLATE完全準拠・完成チェックリスト充足を明示して依頼する） |
| SPEC有り・`docs/spec-qa/<spec名>/` が無い | `/interrogate-spec docs/SPEC.md` でSPECを敵対的レビューする |
| 尋問済み・SPEC §6画面一覧に「(未作成)」がある、または `docs/wireframes/` に対応HTMLが無い | `/wireframe-gen` でワイヤーフレームを生成する |
| `tickets/active/` が0件（.gitkeepを除く） | `/plan-tickets` でSPECからチケットを一括起票し、`/start-loop`（連続実行は `/batch-loop`）でループを開始する。単発の起票は `/new-ticket` |
| activeチケットあり | `/start-loop` でループを再開する |

### 6. 結果報告

各ステップの実施/スキップ結果と「次の一手」を一覧にまとめて報告する。

## Never

- Kit本体の開発時、または人間の明示承認が無い場合に `.git` を削除・再初期化しない（手順1）
- `origin` が `kenesis-loop-kit` を含まないときに切り離し処理を実行しない（既に切り離し済みとみなす）
- 人間の確認なしに `.gitignore` を上書きしない
- SPEC.md・ワイヤーフレーム・チケットをこのコマンドで直接作成・編集しない（手順5の誘導先に委ねる）
- CLAUDE.mdへの書き込みは手順4の略称行の追記のみ。他の節を変更しない

## 使用例

```
/setup
```
