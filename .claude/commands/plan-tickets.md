# /plan-tickets

作成済みのSPEC.mdを基に開発をチケットへ分割し、人間の承認を経て一括起票してください。SPECの要件ID（REQ-xxx）・画面ID（SCR-xxx）・インターフェースID（IF-xxx）とチケットのトレーサビリティ（SPEC_TEMPLATE.md冒頭規約「チケットには対応するIDを明記すること」）を保証するコマンドです。あわせて、非機能要件（NFR-xxx）とエラーハンドリング方針（EH-xxx）を「どのチケットで検証するか」の割当（カバレッジ）も本コマンドで確定します。

## 引数

- 形式: `[spec-path]`（省略時 `docs/SPEC.md`）

## 用語

- **チケット** — 分割の単位。1チケット＝1回の実装ループで回りきる大きさとする
- **マイルストーン** — 依存順に並べたチケットのまとまり（提案時のグルーピング表示に使う）。architectがチケット内部で行う「Phase分割」とは別物であり、本コマンドで「フェーズ」という語は使わない

## 手順

### 1. 前提チェック

1. 対象SPECが存在するか確認する。無ければ `/spec-interview` を案内して終了する
2. `docs/spec-qa/<spec名>/` の有無を確認する。無い場合は「SPEC尋問が未実施のため分割の質が下がる可能性がある」ことを警告し、`/interrogate-spec` を先に実行するか、このまま続けるかを人間に確認する

### 2. SPECの読み取りと既存チケットの突き合わせ（冪等性）

1. SPECの §5 機能要件（REQ・MoSCoW分類）・§6 画面・インターフェース一覧（SCR/IF・REQ対応）・§7 エラーハンドリング方針（EH・検証方法）・§8 非機能要件（NFR・検証方法）・§11 スコープ外・§12 未決事項（OQ）・§13 成功基準を読み取る
2. tickets/active/ と tickets/done/ の全チケットを走査し、本文に記載されたREQ-IDから**カバー済みREQ**を特定する。NFR-ID・EH-IDも同様に走査し、**割当済みNFR/EH**を特定する
   - **`status: cancelled` のチケットは集計から除外する**（不要と判断してクローズしたチケットのREQは実装されていないため。cancelledはdone/に保管されるので見落とさないこと）
3. 未カバーのREQと未割当のNFR/EHのみを分割・割当対象とする（再実行時はSPEC改訂後の差分起票として機能する）
4. **未カバーREQ・未割当NFR/EHがいずれも0件の場合**は起票せず、以下を報告して終了する
   - REQごとのカバレッジ表（REQ-ID / 担当チケットID / チケットstatus）とNFR/EHごとの割当表（NFR/EH-ID / 担当チケットID または「人間が確認（ループ外）」）を提示し、SPECの全要件（少なくとも全Must）がカバー済みであることを示す
   - 次の選択肢を人間に案内する
     - 成果物に見直したいものがあれば `/improvement-loop {チケットID}` で改善ループを起動する
     - 次の開発サイクルへ進むなら `/spec-interview` でSPECを改訂する（改訂後に `/plan-tickets` を再実行すると差分起票される）
     - tickets/done/ が20件を超えていれば `/archive` を提案する

### 3. 分割案の作成

- **粒度基準**: 1チケット＝1回の実装ループで完了する大きさ。受け入れ条件が3〜7個程度で書ける単位を目安とする
- REQ 1件＝1チケットの機械的な分割はしない。画面・機能のまとまりで関連REQを束ねる
- **priority** はMoSCoWからマッピングする: Must→`high`、Should→`medium`、Could→`low`（複数REQが混在するチケットは最上位に合わせる）
- **依存関係**: 技術的な前後関係（例: 認証基盤→認証を要する画面）から起票順を決め、依存先チケットを明示する
- **除外**: §11 スコープ外のREQは起票しない。OQに依存するREQ（例: 「OQ-002に依存」の注記）は保留一覧に分離し、起票するかは人間に確認する
- **NFR/EHの割当**: 検証方法が自動検証のNFR・EHは、最も関連する機能チケットの受け入れ条件に含めるか、まとめて検証する横断検証チケットを新設して割り当てる。検証方法が「人間が確認」の項目はチケットに割り当てず、「人間が確認（ループ外）」として手順4の割当表に明示する

### 4. 分割案の提示と人間の承認（必須）

以下を提示し、人間の承認を得るまで起票しない。修正指示があれば案を更新して再提示する。

1. **分割案テーブル**: 起票順 / タイトル / 対象REQ / 対象SCR・IF / priority / 依存チケット / 受け入れ条件ドラフト（SPECのREQ文言と§13成功基準から作成）
2. **保留一覧**: OQ依存で起票を保留したREQとその理由
3. **カバレッジ報告**: どのチケットにも割り当てなかったREQ（特にMust）があればその理由
4. **NFR/EH割当表**: NFR/EHごとに担当チケット（既存チケットへの追加割当・新設の横断検証チケットを含む）、または「人間が確認（ループ外）」を提示する。全NFR/EHがいずれかに割り当てられていること（宙に浮いたIDを残さない）

### 5. 一括起票

承認後、各チケットを起票順に作成する。作成手順は `/new-ticket`（new-ticket.md）と同じ規約に従う。

- プロジェクト略称の特定 → `tickets/active/` と `tickets/done/` から連番採番 → `tickets/Templates/ticket.md` をコピーして置換 → `tickets/active/{ID}_{タイトル}.md` として**Writeツールで**保存（Bashリダイレクト禁止）
- **概要**に対象REQ-ID・SCR-ID（UIを持たない要件はIF-ID）と、割り当てたNFR-ID・EH-IDを明記する
- **受け入れ条件**に承認済みドラフトを記載する（割り当てたNFR/EHの検証方法を含める）
- **関連**セクションに「依存: {チケットID}」と対象SPEC（`docs/SPEC.md` §5該当行）を記載する

### 6. 完了報告と次の一手

1. 起票したチケットの一覧（ID・タイトル・priority・依存）を依存順で表示する
2. ループの開始方法を案内する: 1件ずつ進めるなら `/start-loop {先頭ID}`、承認ゲートで止めずに連続実行するなら `/batch-loop {依存順のID列}`

## Never

- 人間の承認なしに起票しない
- SPEC.mdを編集しない（SPECの不備を見つけたら `/interrogate-spec` または `/spec-interview` を案内する）
- §11 スコープ外の要件を起票しない
- 既存チケットがカバーするREQを重複起票しない
- REQ 1件＝1チケットの機械的な分割をしない
- 検証方法が「人間が確認」のNFR/EHをチケットに割り当てない（ループ外の人間確認として提示する）

## 使用例

```
/plan-tickets
/plan-tickets docs/SPEC.md
```
